## Основная информация

Реализация Universo Platformo / Universo MMOOMM / Universo Kiberplano на основе Actix Web, Yew и экосистемы Rust. Этот проект является реализацией концепций из [Universo Platformo React](https://github.com/teknokomo/universo-platformo-react) на родном Rust, с фокусом на производительность, безопасность и возможности фронтенда на базе WebAssembly.

**В этом репозитории в настоящее время общественными силами создаётся Universo Platformo / Universo MMOOMM, чтобы запустить глобальную текнокомизацию и спасти человечество от окончательного порабощения и полного уничтожения путём создания специальных массовых многопользовательских виртуальных миров, таких как Universo MMOOMM, и платформы для их создания - Universo Platformo, изначально с игровой функциональностью, а затем с добавлением функциональности Киберплана.**

Universo Platformo Rust служит основой для реализации **Universo Kiberplano** - глобальной системы планирования и реализации, которая объединяет планы, задачи и ресурсы, контролируя роботов. Эта реализация на Rust приоритизирует типобезопасность, производительность и надёжность для критически важных операций.

Более подробно обо всём этом написано в "Книге Будущего" и различных других наших материалах, большинство из которых пока плохо структурированы и не на английском языке, но прямо сейчас ведётся работа по созданию новой подробной документации, которая будет представлена на многих языках.

## Обзор

Universo Platformo Rust — это проект, который переосмысливает основные концепции из Universo Platformo React, используя:

-   **Rust + WebAssembly** для высокопроизводительного, типобезопасного фронтенда (через Yew)
-   **Actix Web** для быстрого, конкурентного бэкенда
-   **Интеграцию Supabase** для аутентификации, доступной исключительно через бэкенд
-   **Структуру монорепозитория** с использованием Cargo workspaces
-   **Архитектуру на основе пакетов** с чётким разделением фронтенда/бэкенда

Проект нацелен на создание единой платформы для разработки интерактивных 3D-приложений, которые могут быть экспортированы в различные технологии, включая AR.js, PlayCanvas, Babylon.js, Three.js и A-Frame, всё на основе гарантий производительности и безопасности Rust.

## Архитектура безопасности

**Supabase доступен исключительно через бэкенд.** Фронтенд никогда не общается с Supabase напрямую — все операции аутентификации и работы с данными проходят через бэкенд Actix Web:

```
Браузер (Yew/WASM) ──► Бэкенд Actix Web ──► Supabase
       │                      │
       └── cookie сессии     └── JWT (в зашифрованной HttpOnly cookie)
```

Это обеспечивает:

-   JWT токен Supabase никогда не попадает в JavaScript браузера
-   Вся логика аутентификации выполняется на стороне сервера
-   Данные сессии хранятся в зашифрованных куках с флагами HttpOnly и SameSite=Lax
-   Ключ `anon_key` Supabase остаётся только на сервере

## Связь с Universo Platformo React

Этот репозиторий реализует ту же концептуальную основу, что и [Universo Platformo React](https://github.com/teknokomo/universo-platformo-react), но с использованием технологического стека Rust:

**Концептуальные эквиваленты:**
-   React → Yew (фреймворк фронтенда на основе WebAssembly)
-   Express → Actix Web (высокопроизводительный веб-фреймворк)
-   PNPM workspaces → Cargo workspaces
-   JavaScript/TypeScript → Rust
-   Supabase JS клиент → HTTP клиент reqwest (только на сервере, никогда в браузере)

**Важно:** Это НЕ прямой перенос. Мы берём проверенные концепции из реализации на React и применяем лучшие практики экосистемы Rust, избегая любых архитектурных проблем или незавершённых функций из версии на React.

## Технологический стек

**Фронтенд (start-frt):**
-   Rust (stable), компилируемый в WebAssembly
-   Yew 0.21 (компонентный UI-фреймворк)
-   Yew Router 0.18 (клиентская маршрутизация)
-   gloo-net (HTTP fetch API с поддержкой credentials для WASM)
-   wasm-bindgen / wasm-bindgen-futures (интеграция с JavaScript)
-   Trunk (сервер разработки с горячей перезагрузкой и API-прокси)

**Бэкенд (start-srv):**
-   Rust (stable)
-   Actix Web 4 (асинхронный HTTP-фреймворк)
-   actix-session 0.9 с CookieSessionStore (зашифрованные/подписанные cookie-based сессии)
-   actix-cors 0.7 (настраиваемый CORS middleware)
-   reqwest 0.12 (асинхронный HTTP-клиент для вызовов REST API Supabase)
-   dotenvy (загрузка переменных окружения из .env)

**Поток аутентификации:**
1. Браузер → `POST /api/v1/auth/login` → Бэкенд → Supabase `/auth/v1/token`
2. Бэкенд сохраняет JWT в зашифрованной куке сессии: HttpOnly, SameSite=Lax
3. Браузер → `GET /api/v1/auth/me` → Бэкенд → Supabase `/auth/v1/user`
4. Браузер → `POST /api/v1/auth/logout` → Бэкенд → Supabase `/auth/v1/logout` + удаление сессии

**Инструменты разработки:**
-   Cargo workspaces (управление монорепозиторием)
-   rustfmt (форматирование кода)
-   clippy (линтинг)
-   wasm-bindgen-test (юнит-тесты WASM)
-   actix-web::test (интеграционные тесты бэкенда)

## Структура проекта

**КРИТИЧЕСКИ ВАЖНО: Модульная архитектура (ОБЯЗАТЕЛЬНА)**

ВСЯ функциональность приложения ДОЛЖНА быть реализована как пакеты в директории `packages/`. Это фундаментальное архитектурное требование, которое не может быть нарушено.

```
universo-platformo-rust/
├── packages/                  # ВЕСЬ КОД ПРИЛОЖЕНИЯ ДОЛЖЕН БЫТЬ ЗДЕСЬ
│   ├── start-frt/             # Пакет фронтенда стартовой страницы
│   │   └── base/
│   │       ├── src/
│   │       │   ├── api/       # HTTP-клиент (gloo-net, fetch с credentials)
│   │       │   ├── auth/      # AuthContext, AuthProvider, состояние use_reducer
│   │       │   ├── components/# Hero, LoginForm, OnboardingWizard, Footer и др.
│   │       │   ├── pages/     # StartPage, GuestStartPage, AuthenticatedStartPage
│   │       │   ├── app.rs     # Корневой компонент App с Yew Router
│   │       │   └── lib.rs     # Точка входа WASM (wasm_bindgen start)
│   │       ├── index.html     # HTML точка входа для Trunk
│   │       ├── styles.css     # Стилевая таблица с тёмной темой
│   │       └── Trunk.toml     # Конфиг Trunk с API-прокси на бэкенд
│   └── start-srv/             # Пакет бэкенда стартовой страницы
│       └── base/
│           ├── src/
│           │   ├── handlers/  # auth.rs (login/logout/me), onboarding.rs
│           │   ├── config.rs  # AppConfig из переменных окружения
│           │   ├── error.rs   # Перечисление AppError с маппингом HTTP-ответов
│           │   ├── models.rs  # Общие типы данных запросов/ответов
│           │   ├── supabase.rs# REST API клиент Supabase (sign_in/get_user/sign_out)
│           │   └── main.rs    # Настройка HttpServer, маршрутизация, middleware
│           └── .env.example   # Шаблон документации переменных окружения
├── Cargo.toml                 # Определение workspace и общие зависимости
└── README.md                  # Этот файл
```

**Архитектурные ограничения:**
- ✅ Весь код функций находится в `packages/`
- ✅ Фронтенд вызывает только эндпоинты `/api/v1/*` на бэкенде
- ✅ Бэкенд является единственным клиентом REST API Supabase
- ❌ Никаких прямых вызовов Supabase из фронтенда (браузер не касается Supabase)
- ❌ Никакого кода приложения в корне репозитория (без директории `src/`)

## Ключевые функции

### Аутентификация и стартовая страница

Первая реализованная функция обеспечивает полный цикл аутентификации:

-   **Лендинг для гостей**: Раздел героя, отзывы, кнопки "Войти" / "Начать"
-   **Модальное окно входа**: Форма email/пароль с вызовом эндпоинта аутентификации бэкенда
-   **Управление сессией**: JWT хранится в зашифрованной HttpOnly cookie (cookie-based session)
-   **Панель авторизованного пользователя**: Трёхшаговый мастер онбординга после входа
-   **Восстановление сессии**: При каждой загрузке `GET /api/v1/auth/me` восстанавливает сессию
-   **Выход**: Отзывает токен Supabase и очищает зашифрованную cookie сессии

### Трёхшаговый мастер онбординга

Мастер онбординга следует паттерну трёх сущностей, используемому во всём Universo Platformo:

-   **Шаг 1 — Проекты** (Глобальные цели): Инициативы устойчивого развития
-   **Шаг 2 — Кампании** (Личные интересы): Образование, здравоохранение, экология
-   **Шаг 3 — Кластеры** (Функции платформы): 3D-миры, AR-опыты, VR-симуляции
-   **Экран завершения**: Отображается после отправки мастера, с возможностью обновить настройки

### Планируемые функции

-   **Кластеры**: Полный CRUD для сущностей Кластеры / Домены / Ресурсы
-   **Метавселенные**: Управление Метавселенными / Секциями / Сущностями
-   **Пространства/Холсты**: Визуальный редактор рабочих процессов на основе узлов для WebAssembly
-   **Общие пакеты**: `universo-types`, `universo-utils`, `universo-api-client`

## Рабочий процесс разработки

1. **Фаза планирования:**
   - Создать GitHub Issue (следуя `.github/instructions/github-issues.md`)
   - Написать спецификацию функции в `.specify/specs/[###-feature-name]/spec.md`
   - Получить одобрение спецификации

2. **Фаза реализации:**
   - Создать ветку функции с именем `###-feature-name`
   - Реализовать согласно спецификации
   - Писать тесты вместе с кодом
   - Обновить документацию (сначала английскую, затем русскую)

3. **Контрольные точки качества:**
   - Весь код должен проходить `cargo clippy` без предупреждений
   - Весь код должен быть отформатирован с `rustfmt`
   - Все тесты должны проходить с `cargo test`
   - Документация должна быть двуязычной с идентичной структурой

4. **Проверка и слияние:**
   - Создать Pull Request (следуя `.github/instructions/github-pr.md`)
   - PR должен ссылаться на соответствующий Issue
   - PR должен включать описания на английском и русском языках
   - PR должен пройти все проверки CI перед слиянием

## Начало работы

### Предварительные требования

-   Rust (stable) — Установить через [rustup](https://rustup.rs/)
-   Trunk — `cargo install trunk`
-   Цель wasm32 — `rustup target add wasm32-unknown-unknown`
-   Аккаунт Supabase — [supabase.com](https://supabase.com) с существующей учётной записью пользователя

### Запуск бэкенда

```bash
# Перейти в директорию пакета бэкенда
cd packages/start-srv/base

# Скопировать шаблон окружения и заполнить значениями
cp .env.example .env
# Редактировать .env: установить SUPABASE_URL и SUPABASE_ANON_KEY

# Запустить сервер Actix Web на http://127.0.0.1:8080
cargo run
```

### Запуск фронтенда

```bash
# Перейти в директорию пакета фронтенда
cd packages/start-frt/base

# Запустить сервер разработки Trunk (проксирует /api на localhost:8080)
trunk serve
```

Откройте `http://127.0.0.1:8081` в браузере. Бэкенд (порт 8080) и сервер Trunk (порт 8081) должны работать одновременно.

### Сборка для продакшена

```bash
# Собрать бинарный файл бэкенда
cargo build --release -p start-srv

# Собрать WASM-бандл фронтенда
cd packages/start-frt/base && trunk build --release
```

### Создание нового пакета функции

При создании новой функции, требующей фронтенд и бэкенд:

1. Создать два пакета в `packages/`:
   ```bash
   mkdir -p packages/myfeature-frt/base
   mkdir -p packages/myfeature-srv/base
   ```

2. Добавить их в workspace в `Cargo.toml`

3. Разместить код Rust в соответствующих директориях `base/`

4. Следовать подходу specification-first, задокументированному в `.specify/`
